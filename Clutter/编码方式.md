# Unicode和ASCII
Unicode 和 ASCII 都是字符编码标准，但它们在设计目标、容量和实现方式上有本质区别。简单来说：ASCII 是 Unicode 的一个极小子集，而 Unicode 是涵盖全球所有书写系统的“大一统”方案。

## 介绍
            ASCII	            Unicode
全称	美国信息交换标准代码	统一码 / 万国码
设计年份	1960年代	        1990年代
包含字符数	128 个（95个可打印字符+33个控制符）	超过 14.9万 个（且持续增加）
涵盖语言	仅基础英文 + 数字 + 符号	全球所有现代文字、古代文字、Emoji 表情
占用字节	1 字节（7位有效）	变长（UTF-8为1-4字节，UTF-16为2或4字节）
英文存储效率	高	            完全兼容 ASCII，英文部分也是 1 字节

## 场景分析
ASCII 能做什么？
只能写纯英文的 Hello, World! 或 12345。

ASCII 做不到而 Unicode 能做到的：
显示汉字 “中文”、显示德语 “München”（带分音符的 ü）、显示 Emoji 😊。

## Unicode 包含 ASCII？
Unicode 在设计时特意保留了对 ASCII 的完全兼容。
编码层面：Unicode 的前 128 个码位（U+0000 到 U+007F）就是 ASCII 码。
实际效果：一个用 UTF-8 编码保存的纯英文 .txt 文件，其二进制内容与用 ASCII 保存的完全一样。

补充： 
UTF-8和UTF-16的区别
对比维度	UTF-8	UTF-16
基本编码单元	1 字节（8位）	2 字节（16位）
长度范围	变长：1 ~ 4 字节	变长：2 或 4 字节
英文/数字存储	1 字节（高效，与 ASCII 完全相同）	2 字节（浪费空间）
中文/日文存储	通常 3 字节	通常 2 字节（对东亚文字略优）
字节序问题	无（单字节序列天然有序）	有（需指明 UTF-16LE 或 UTF-16BE，否则乱码）
网络/文件首选	✅ 互联网事实标准（网页、JSON、API）	极少用于 Web，多用于 Windows 内核、Java 内存
2. 具体存储对比（同一个汉字 “中”）
Unicode 码点：U+4E2D

UTF-8 编码后：E4 B8 AD （3 个字节）

UTF-16 编码后：4E 2D （2 个字节）

3. 关键差异深度解析
A. 对 ASCII 兼容性（这是 UTF-8 统治互联网的根本原因）
UTF-8：一个纯英文 Hello，编码后是 48 65 6C 6C 6F，与 50 年前的 ASCII 码完全一致。老旧设备不认识 Unicode，但能正确读出英文部分。

UTF-16：一个纯英文 Hello，编码后是 00 48 00 65 00 6C 00 6C 00 6F。旧设备读到第一个字节 00 可能直接认为文件结束了（造成 C 语言字符串截断）。

B. 字节序（Byte Order Mark, BOM）
UTF-8：没有大小端争议。E4 永远在 B8 前面，不需要 BOM 头（加 EF BB BF 只是为了声明这是 UTF-8，不影响解析）。

UTF-16：极度依赖字节序。4E 2D 还是 2D 4E？如果 Windows 保存时写成了 2D 4E，Linux 按 4E 2D 读，就会变成乱码 “⸭” 而非 “中”。这就是为什么你打开某些文本文件开头会看到 FF FE（UTF-16 LE BOM 头）的原因。

C. 存储效率的取舍
如果你主要写英文：UTF-8 省一半空间（1字节 vs 2字节）。

如果你主要写中文、日文、韩文：UTF-16 省 1/3 空间（2字节 vs 3字节）。

4. 为什么现实中 UTF-8 赢了？

总结一句口诀：

UTF-8 是互联网普通话，UTF-16 是 Java/Windows 方言。
